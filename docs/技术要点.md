# RAG 论文知识问答系统 — 技术架构白皮书

## 系统架构总览

```
用户查询
    │
    ▼
┌──────────────────────────────┐

           ▼
┌──────────────────────────────┐
│      Dual-Retrieval          │
│  ┌────────────────────┐      │
│  │ Dense Vector (BGE) │      │  ← 语义检索 (1024维)
│  └────────┬───────────┘      │
│  ┌────────┴───────────┐      │
│  │ Sparse BM25        │      │  ← 关键词精确匹配
│  └────────┬───────────┘      │

└──────────┬───────────────────┘
           │
           ▼

           ▼

│    └────────────────────┘    │
└──────────────────────────────┘
```

## 核心技术要点

### 1. 混合检索（Hybrid Search）

采用双通道检索 + RRF 融合排序策略，兼顾语义理解与关键词精确匹配：

| 通道 | 模型 | 维度 | 特点 |
|------|------|------|------|
| 稠密向量 | BAAI/bge-large-zh-v1.5 | 1024 | 语义相似度，跨语言泛化 |
| 稀疏向量 | BM25 算法 | 词汇表大小 | 关键词精确命中，OOV 鲁棒 |

**RRF 融合公式：**

```
score(d) = 1 / (k + rank_dense(d)) + 1 / (k + rank_sparse(d))
```

其中 `k=60` 为平滑常数，兼顾两通道的排名置信度。融合后产生最终排序列表。

### 2. CrossEncoder 重排序（Reranker）

初筛 Top-K 后，引入 BGE-reranker-base 进行二次精排：

- **联合编码**：将 query 与每个候选文档拼接后输入 Transformer，计算相关性得分
- **精度优势**：相比向量余弦距离（双塔）的近似匹配，CrossEncoder（单塔）精度显著更高
- **效率优化**：仅对 Top-50 候选重排，在精度与延迟之间取得平衡

### 3. HyDE 查询扩展（Hypothetical Document Embeddings）

针对查询语义与文档语义之间的分布偏移，引入 HyDE 策略：

1. 将原始查询送入 LLM，生成一段"假设文档"（Hypothetical Document）
2. 对假设文档做 embedding，用其向量检索知识库
3. 假设文档的语义分布更接近真实文档，从而提升召回质量

**适用场景**：短查询、模糊查询、术语不一致的跨领域检索。

### 4. 语义分块策略（Semantic Chunking）

预处理阶段对论文 Markdown 进行智能分块：

- **边界感知**：以句子为基本单位，避免在句中断开
- **上下文保留**：每个块携带所在小节的标题层级信息（`#` / `##` / `###`）
- **块大小**：约 1500 字符，兼顾语义完整性与检索粒度
- **去重机制**：入库阶段基于内容哈希自动跳过已存在的段落

### 5. 模糊问题检测（Ambiguity Detection）

系统具备主动的意图澄清能力：

1. 用户提交查询后，LLM 评估问题是否模糊或过于宽泛
2. 若检测为模糊，不直接检索回答，而是返回 3~4 个具体化的建议方向
3. 用户选择具体方向后，系统再执行完整检索生成流程

**预置领域关键词**：transformer、attention、cnn、rnn、gan、gnn、llm、nlp、reinforcement learning 等 CS 核心方向。

### 6. 流式输出引擎（SSE Streaming）

基于 Server-Sent Events 协议，实现端到端的实时流式响应：

```
消息流时间线：
┌──────────┬─────────────┬──────────────┬────────────┬──────────────┐
│ retrieved │  ambiguous   │  generating  │  finished  │     done      │
│ (检索文档) │ (模糊检测)  │  (逐token)   │ (回答完成) │ (相关问题+    │
│           │             │              │            │  Token统计)   │
└──────────┴─────────────┴──────────────┴────────────┴──────────────┘
```

- **retrieved**：检索到的相关文档段落及相关性分数
- **ambiguous**：可选阶段，仅当检测到模糊查询时触发
- **generating**：LLM 思考过程 + 逐 token 生成内容
- **finished**：回答生成完毕
- **done**：携带相关问题和 Token 使用统计

前端基于 `ReadableStream.getReader()` 实现逐行解析，零依赖纯原生 JS。

### 7. 多轮对话上下文管理

- **持久化**：对话历史存储在浏览器 localStorage，刷新页面不丢失
- **上下文传递**：每次请求携带完整 `chat_history`，支持指代消解与追问
- **并发控制**：单次只能进行一轮对话，防止上下文错乱

### 8. 相关问题智能推荐

回答完成后，LLM 自动根据问答内容生成 2~3 个深度追问：

- **生成时机**：流式 `done` 阶段，此时已拥有完整回答文本
- **生成策略**：基于原始问题与回答摘要（前 600 字符），引导用户深化理解
- **交互方式**：推荐问题以按钮形式展示，点击即发送新查询

### 9. Token 使用量估算

每次问答自动统计 Token 消耗，在前端清晰展示：

```
┌─ Token 统计 ──────────────┐
│  Input:    ~245 tokens     │
│  Output:   ~512 tokens     │
│  Total:    ~757 tokens     │
└────────────────────────────┘
```

- **估算方法**：中文字符 ÷ 4 + 英文字符 ÷ 3（业界通用近似算法）
- **展示时机**：流式 `done` 阶段，与相关问题一同返回
- **用途**：监控 API 消耗，优化查询成本

### 10. 点赞/点踩反馈系统

每轮回答提供评价机制：

- **本地持久化**：反馈数据存储在浏览器 localStorage
- **操作反馈**：点赞/点踩切换，视觉高亮反馈
- **数据留存**：可用于后续评估回答质量或微调

## Milvus 向量数据库部署

支持两种模式，平滑切换：

| 维度 | 本地模式（Milvus Lite） | Docker 模式（Milvus Standalone） |
|------|------------------------|----------------------------------|
| 依赖 | 无 | etcd + MinIO |
| 启动 | 零配置，进程内嵌入 | docker compose up -d |
| 场景 | 开发测试、小规模 | 生产部署、大规模 |
| 连接 | 本地文件 | gRPC :19530 |

Docker 模式完整启动链：

```
etcd (配置中心) ──→ MinIO (向量存储) ──→ Milvus (查询引擎) ──→ RAG App (推理服务)
```

## 性能优化要点

1. **模型缓存**：Host 侧 HuggingFace 缓存挂载到容器，避免重复下载
2. **去重入库**：段落级内容哈希去重，增量更新不重复写入
3. **重排序剪枝**：仅对 Top-50 做 CrossEncoder 精排，控制延迟
4. **并发安全**：RAG 引擎实例加锁，Milvus 连接池复用
5. **Python 缓冲优化**：Docker 环境设置 `PYTHONUNBUFFERED=1`，确保日志实时输出
