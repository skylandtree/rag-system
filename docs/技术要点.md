# RAG 系统技术要点

## 1. 混合检索（Hybrid Search）

采用 **稠密向量 + 稀疏 BM25 双通道检索**，最后用 **RRF（Reciprocal Rank Fusion）** 融合排序：

- **稠密向量**：BGE-large-zh-v1.5 生成 1024 维语义向量，捕捉语义相似度
- **稀疏 BM25 向量**：基于词频统计的关键词匹配，确保精确命中
- **RRF 融合**：`score = 1/(60 + rank_dense) + 1/(60 + rank_sparse)`，兼顾语义与关键词

## 2. 重排序（Reranker）

初步检索 Top-K 后，用 **BGE-reranker-base CrossEncoder** 二次排序：

- 相比向量余弦相似度，CrossEncoder 对 query-doc 对做联合编码，精度更高
- 只对 Top-K（如 50 条）重排，控制计算成本

## 3. HyDE 查询扩展

对模糊查询，先用 LLM 生成一篇"假设文档"（用 LLM 回答该问题），再用假设文档的 embedding 去检索：

- 解决 query embedding 与 document embedding 之间的分布偏移问题
- 显著提升模糊查询的召回质量

## 4. 语义分块

按句子边界感知切分，保留小节标题作为上下文：

- 块大小 ~1500 字符，避免信息碎片化
- 保留 Markdown 标题层级（`#` `##`）作为段落前缀，增强块级语义

## 5. 流式输出（SSE）

Flask 后端以 Server-Sent Events 分阶段推送：

```
retrieved → [ambiguous] → generating / finished → done (+ related_questions)
```

- 前端用 `ReadableStream.getReader()` 逐行读取
- 支持展示检索文档、思考过程、逐 token 生成、相关问题推荐

## 6. Milvus 向量数据库

支持两种部署方式：

- **本地模式**：Milvus Lite（嵌入式），零配置，适合开发测试
- **Docker 模式**：Milvus Standalone（etcd + MinIO），生产级部署

Docker 模式下完整启动链：`etcd（配置中心）→ MinIO（对象存储）→ Milvus（向量引擎）→ RAG App`

## 7. 多轮对话

对话历史以 `chat_history` 参数传入 LLM，支持上下文记忆：

- 前端用 localStorage 持久化，刷新不丢失
- LLM 回答时附带历史消息，实现追问、指代消解

## 8. 模糊问题检测

LLM 判断用户问题是否模糊，若是则返回澄清建议而非直接回答：

- 匹配预定义的 CS 领域关键词（transformer、cnn、rnn 等）
- 让用户选择具体方向后再检索，避免宽泛问题返回无用结果
